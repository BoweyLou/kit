# Agentic Coding Regression Findings

Generated: 2026-05-06

## Executive Readout

The three raw datasets point to the same failure mode from different angles:
agentic coding regressions are rarely just "the model got worse." They are
usually a control-system problem: weak context packing, stale or conflicting
instructions, unbounded tool permissions, missing red/green evidence, noisy
review automation, and no durable receipt that proves what an agent actually
read, changed, skipped, and verified.

The practical direction for `agent-workflow-kit` and `repo-contract-kit` should
therefore be conservative and local-first:

1. Treat agent instructions as executable policy, not prose.
2. Treat every agent run as an auditable transaction with a receipt.
3. Make read-only review the default and write-capable work explicitly scoped.
4. Add regression fixtures for prompts, review findings, docs-impact detection,
   and test-first claims before adding more autonomy.
5. Borrow external patterns where they are implementation examples, but build
   the core governance layer locally because the user's problem is trust,
   repeatability, and repo-specific control.

## Dataset Analysis

### X

Source: `2026-05-05-x-raw.md`

The X dataset has 123 authenticated read-only browser-search results. Its strongest signal is
not vendor-grade evidence; it is practitioner language around felt regressions:
model/version regressions, agentic coding getting worse despite benchmarks,
token burn, long-context reliability, brittle prompt changes, test generation,
and frustration with false-positive review output.

Actionable signals:

- Evals need to be workflow-specific. General model benchmarks do not predict
  whether an agent will preserve repo intent, choose the right tool, or avoid
  repeating a fixed mistake.
- Test generation is a real productivity win only when red/green evidence and
  generated-test provenance are recorded.
- Prompt tweaks create hidden regressions unless prompts have fixtures and
  expected outputs.
- "Vibe" assessment is too weak for sustained agentic coding; we need receipts,
  outcome tracking, and replayable examples.

Limits:

- X is anecdotal and personalized.
- Some entries are model-release commentary rather than concrete repo workflow
  failures.
- Treat X as a vocabulary and pain-radar source, not as validation.

### GitHub

Source: `2026-05-05-github-raw.md`

The GitHub dataset has 130 GitHub URLs. Because GitHub search hit secondary
rate limits, the captured dataset is skewed toward issues and pull requests
matching AI code-review false-positive terms. That skew is still valuable: it
shows that the code-review automation problem is not "can an agent comment?"
but "can we prevent low-signal output from becoming review debt?"

Actionable signals:

- False-positive handling must be first-class: severity, confidence, evidence,
  why-this-might-be-wrong, and accepted/rejected disposition.
- Review bots need finding budgets and no-nit defaults.
- PR automation should emit structured artifacts before comments.
- GitHub itself is not the control plane; it is one adapter over local checks,
  receipts, and safe-output JSON.

Limits:

- The dataset over-represents PRs matching search terms.
- It is useful for noise and review-process design, less useful for model
  regression diagnosis.
- Secondary rate-limit warnings mean any implementation borrowing from this
  dataset should be confirmed against specific upstream projects before adoption.

### Hacker News

Source: `2026-05-05-hn-raw.md`

The HN dataset has 125 HN item/comment/story URLs. It is the best source for
architecture and failure-mode framing. The recurring themes are context rot,
MCP/tool orchestration, sandboxing, policies, established-codebase use,
multi-agent consensus, and skepticism about benchmark-only claims.

Actionable signals:

- Context is a product surface. Agents need a repeatable diff-to-context pack
  that includes changed files, callers, tests, docs, ADRs, runtime configs, and
  prior accepted findings.
- Tool permissions need an enforceable policy shape. "Use tools carefully" is
  not enough for file, shell, git, browser, MCP, and CI access.
- Multi-agent setups need role isolation and synthesis, not just more agents.
- Established-codebase work needs comments/docs/ADRs/tests as contract inputs.

Limits:

- HN is also anecdotal, but its discussions are often more explicit about
  architecture and workflow constraints.
- Some HN items point to external projects; those projects should be evaluated
  directly before adoption.

## Cross-Cutting Patterns

1. Context drift is the highest-leverage failure mode.
   Agents lose repo intent when they only see a diff or a large unstructured
   context dump. The kit should provide a deterministic context packet, not rely
   on each agent to rediscover the same files.

2. Instruction drift is now a correctness risk.
   `AGENTS.md`, `CLAUDE.md`, Cursor rules, prompt files, and review policies can
   contradict each other. These files need linting for stale paths, stale
   commands, bloat, hidden Unicode, unsafe permissions, and duplicated rules.

3. Review noise is a workflow regression.
   AI review that produces many low-confidence comments can make teams slower.
   Finding budgets, severity/confidence thresholds, false-positive notes, and
   outcome tracking are not polish; they are required controls.

4. Test-first claims need evidence.
   A prompt saying "use TDD" is insufficient. Receipts should require failing
   command output, passing command output, generated-test provenance, and an
   explicit exception reason when test-first is not feasible.

5. Tool permission is a policy problem.
   MCP servers, browser access, shell, git, network, and CI jobs need a visible
   allowlist and trust model. Read-only reviewers should be unable to mutate the
   repo by contract and by runner behavior.

6. Multi-agent review needs synthesis and isolation.
   More agents can create more duplication. Personas need source boundaries,
   allowed actions, output schemas, max findings, and one severity-ranked
   synthesis report.

7. The missing artifact is a regression harness.
   The repo already has prompts, schemas, receipts, and backlog rows. The next
   major quality jump is fixtures that catch regressions in prompt outputs,
   docs-impact detection, review-finding quality, and instruction linting.

## Build Recommendations By Repo

### `agent-workflow-kit`

Own the agent-facing workflow, prompts, schemas, and regression harness.

Build next:

1. Prompt regression fixture suite.
   Add golden examples for review synthesis, TDD receipts, learning-comments,
   and task packets. Validate output schema, finding budget, evidence fields,
   false-positive notes, and final disposition.

2. Diff-to-context packet guidance.
   Turn AGW-032 into a concrete artifact shape: changed files, likely callers,
   tests, docs, ADRs, scripts, runtime configs, dependency manifests, and recent
   accepted findings.

3. Review outcome ledger schema.
   Extend receipt/synthesis outputs so findings can be marked accepted,
   rejected, duplicate, false positive, fixed, stale, or promoted-to-policy.

4. Read-only reviewer policy.
   Make AGW-040 concrete: reviewer personas cannot write files, run formatters,
   stage changes, or mutate browser/account state. Write-capable workers need
   separate worktrees and explicit acceptance criteria.

5. Browser research agent policy.
   Finish AGW-050 using the X/Safari run as the reference case: source-scoped,
   read-only, no account mutation, no CAPTCHA bypass, and source-count receipts.

### `repo-contract-kit`

Own installed repo guardrails, policy checks, local commands, CI adapters, and
safe output artifacts.

Build next:

1. Agent-instruction lint expansion.
   Finish AGW-002/003/004: stale paths, stale commands, bloat, contradictions,
   hidden Unicode, unsafe network/tool guidance, wildcard permissions, and rule
   provenance.

2. Safe-output and strict receipt validation.
   Extend AGW-026/027 so local and CI runs can reject malformed or missing
   evidence before humans read a report.

3. Fork-safe read-only review profile.
   Implement AGW-005 as a local-first trust profile first, then add optional
   GitHub Action wiring. Untrusted PRs should only produce safe JSON artifacts.

4. Docs patch proposal mode.
   Build AGW-019 so agents can write proposed docs patches under an artifact
   directory instead of directly editing docs by default.

5. Agent policy exporter.
   Advance AGW-012 with generated projections for `AGENTS.md`, `REVIEW.md`,
   `CLAUDE.md`, `.cursor/rules`, `.continue/rules`, and `.windsurf/rules`.
   Add sync checks so runtime-specific files cannot drift.

## Open Questions And Validation Experiments

1. Replay recent local failures.
   Take 5 real Codex/agent regressions from this repo or related repos and see
   whether a receipt validator plus instruction linter would have caught them.

2. Sample GitHub false positives.
   Manually review 20 entries from the GitHub raw report and classify whether
   they represent true false positives, noisy search matches, or useful process
   examples. Use the result to calibrate finding budgets.

3. Test prompt fixture stability.
   Create 10 small fixture tasks and run the same prompt through two model
   versions or two agents. Track schema validity, evidence quality, and finding
   usefulness.

4. Simulate untrusted PR mode.
   Run a read-only review against a dirty branch and confirm the runner cannot
   edit files, stage files, comment on GitHub, or access disallowed network
   targets.

5. Compare context packets.
   Run the same review with raw diff only, broad repo dump, and deterministic
   diff-to-context packet. Compare false positives and missed findings.

## Bottom Line

Do not adopt a full external review bot as the core answer. The stronger move is
to build a local governance spine and borrow selected patterns:

- PR-Agent/Qodo for command vocabulary.
- CodeRabbit for feedback/outcome memory concepts.
- Aider/OpenHands/SWE-agent for git/worktree/task isolation ideas.
- MCP policy projects for permission framing.
- TDD Guard-style checks for red/green proof.

The next implementation slice should be a regression-control slice, not a new
agent feature: instruction linting, receipt validation, prompt fixtures,
read-only reviewer policy, and generated context packets.
