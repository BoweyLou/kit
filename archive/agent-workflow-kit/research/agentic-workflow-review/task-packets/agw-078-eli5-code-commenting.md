## Acceptance Criteria

- The prompt distinguishes "learning developer" support from "non-developer explainability" without assuming the reader will edit code. Verify by reading `workflows/prompts/codebase-learning-comments.md`.
- The workflow requires agents to choose between inline comments and a separate explanation note, and to prefer the note when inline comments would make maintained source noisy. Verify by reading the prompt output section.
- The workflow defines high-value non-developer explanation targets: domain terms, business or research rules, data flow, external boundaries, state transitions, operational constraints, and surprising guardrails. Verify by reading the evidence/comment rules.
- The workflow rejects low-value ELI5 output: syntax narration, framework tutorials, obvious comments, unverifiable simplifications, anthropomorphic explanations, and jargon swaps that lose accuracy. Verify by reading the rejection rules.
- The final diff remains prompt/docs/backlog only, plus generated adapter output. Verify with `git diff --name-only`.
- The generated Codex adapter matches the source prompt. Verify with `make prompt-adapters-check` or `make validate`.
- Documentation impact is covered in user-facing prompt docs or explicitly waived with a concrete reason. Verify with `make docs-check` and `python3 scripts/check_doc_impact.py`.
